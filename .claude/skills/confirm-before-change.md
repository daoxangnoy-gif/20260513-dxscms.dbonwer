# confirm-before-change

ทุกครั้งที่ได้รับคำสั่งให้แก้ไข เพิ่ม หรือลบ code — **ต้องรายงานแผนก่อนเสมอ ห้ามแก้โค้ดทันที**

## รูปแบบที่ต้องแสดงก่อนลงมือแก้

```
📋 แผนการแก้ไข

🔧 ไฟล์ที่แก้:
  - [ชื่อไฟล์:บรรทัด] — สิ่งที่จะเปลี่ยน

🍽️ เมนู / หน้าที่กระทบ:
  - เมนู > หน้า > action ที่เปลี่ยน

📉 ก่อนแก้: พฤติกรรมปัจจุบันเป็นอย่างไร (ระบุตัวเลขหรือผลลัพธ์ที่เห็นได้)
📈 หลังแก้: พฤติกรรมใหม่จะเป็นอย่างไร (ระบุตัวเลขหรือผลลัพธ์ที่คาดหวัง)

⚠️ ความเสี่ยง: [ต่ำ / กลาง / สูง] — เหตุผล

---
✅ ยืนยันให้แก้ได้เลยไหมครับ?
```

## กฎสำคัญ

1. **ห้ามแก้โค้ดก่อนได้รับการยืนยัน** — แสดงแผนแล้วรอ user ตอบกลับก่อนเสมอ
2. **ระบุเมนูให้ชัด** — บอกว่า user ต้องไปที่เมนูไหน กดอะไร ถึงจะเห็นความเปลี่ยนแปลง
3. **ระบุ action ให้ชัด** — เช่น "กด Export", "เปลี่ยน page", "เปิด Tab Show & Edit"
4. **เปรียบ before/after ด้วยตัวเลขหรือพฤติกรรมที่เป็นรูปธรรม** — ไม่ใช่แค่ "เร็วขึ้น" แต่ระบุว่า "จาก 5 query เหลือ 1 query"
5. **ถ้าแก้หลายไฟล์** — ระบุทุกไฟล์ในรายการเดียวกัน อย่าแยกรายงาน

## ข้อยกเว้น (ไม่ต้องรอยืนยัน)

- งานที่ไม่แก้โค้ด เช่น อ่านไฟล์ ดู git log วิเคราะห์
- auto-push หลังจาก user ยืนยันและแก้เสร็จแล้ว
